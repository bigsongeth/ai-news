# AI Daily Digest Workflow 与排障

这份文档讲的是这条技能的业务工作流。

如果 `SKILL.md` 是“这是什么、为什么值得装”，
那么这里就是“具体怎么跑、出问题先看哪层”。

---

## 1. 总体结构

这条流水线建议按“主任务 + 子任务”来理解：

- 主任务：调度、等待、收口、最终输出
- 子任务：Scout / Verifier / Extractor / Editor

一句话：

> 子任务负责产物，主任务负责结果。

---

## 2. 标准阶段

### 2.1 Fetch

作用：

- 从 RSS 或既定信息源抓候选文章

典型产物：

- `output/candidates.json`

常见问题：

- RSS 源失效
- 候选数太少
- 候选数太多导致后面成本变高

### 2.2 Plan

作用：

- 去重
- 过滤
- 分 chunk
- 生成本次 runId 的运行目录

典型产物：

- `output/<runId>/plan.json`
- `chunk-<n>.input.json`

常见问题：

- shortlist 太松，导致后面处理量过大
- shortlist 太紧，导致最后信息面太窄

### 2.3 Scout

作用：

- 判断哪些候选值得优先进入日报
- 形成初步信号与优先级

典型产物：

- `signals.json`

常见问题：

- 过于依赖标题判断
- 引语帖、评论帖权重过高

### 2.4 Verifier

作用：

- 尽量找到更原始、更官方、更权威的来源

典型产物：

- `validated_sources.json`

常见问题：

- 二手媒体链接没被替换成原始来源
- 多条新闻其实指向同一个源头

### 2.5 Extractor

作用：

- 读取文章正文
- 尽量避免只用标题/简介生成摘要

典型产物：

- `chunk-<n>.extracted.json`

硬要求：

- 必须记录 `attempts`
- 必须记录 `failure_stage`
- 必须记录 `failure_reason`
- 必须记录 `content_chars`

常见问题：

- 抓到的是导航页
- 页面过短
- 反爬
- 网页正文拿不全

### 2.6 Editor

作用：

- 把正文内容转成中文展开版摘要
- 补充推荐理由、关键词、评分等结构化信息

典型产物：

- `chunk-<n>.edited.json`

硬要求：

- `summaryZh` 必须是展开版中文总结
- 不能只是标题改写
- 不能直接拼正文原文
- 如果正文不完整，必须明确保守

常见问题：

- 输出统一模板
- 全部是空洞套话
- 像“我读到了，但我不确定”的机械废话

### 2.7 Render

作用：

- 把编辑结果拼成最终日报
- 输出聊天友好的最终格式

典型产物：

- `final.txt`
- 最终 digest 正文

常见问题：

- Top 0
- 排版混乱
- 信息密度太低

---

## 3. 为什么这样拆

这条技能这样拆，不是为了复杂而复杂，而是为了把问题切开：

- 候选质量问题，不要甩给最后一步模型
- 来源可信度问题，不要甩给摘要模型猜
- 正文抓取问题，不要假装总结时已经解决
- 最终排版问题，也不要影响中间质量判断

拆开后的好处：

- 更容易并行
- 更容易替换单层实现
- 更容易定位问题
- 更适合长期运营

---

## 4. 子任务设计原则

### 原则 1：子任务只负责产物

子任务应该：

- 落文件
- 产出结构化结果
- 在设计允许的情况下返回 `ANNOUNCE_SKIP`

子任务不应该：

- 抢着发最终日报
- 对外输出半成品

### 原则 2：主任务负责收口

主任务应该：

- 等待必要文件
- 判断是否继续渲染
- 输出最终正文或明确错误报告

### 原则 3：不要把所有问题都寄希望于一个模型

这条技能的思路不是：

- “给一个超长 prompt，让一个模型全做完”

而是：

- 先把信息准备好
- 再让最后一步做真正该做的编辑与汇总

---

## 5. 排障顺序

### 没候选

优先看：

- fetch
- RSS 可用性
- `candidates.json`

### 来源不准

优先看：

- verifier
- `validated_sources.json`

### 正文为空或太短

优先看：

- extractor
- `failure_reason`
- `content_chars`

### 摘要质量差

优先看：

- editor
- `summaryZh`
- `confidence`
- `score`

### Top 0

优先看：

- editor 输出是否模板化
- render 过滤规则是否过严

### 没发出去

优先看：

- cron run 状态
- delivery 配置
- 目标渠道是否可用
- 是否存在旧 cron / 脏 session / 旧 fallback 污染

---

## 6. 一个很重要的经验

如果旧 cron 一直抽风，不要一开始就死磕旧 job。

实践上更稳的顺序通常是：

1. 先确认是“没触发”还是“触发后失败”
2. 再区分“配置问题”还是“运行时问题”
3. 如果旧 job 明显被污染，直接新建一条干净 cron
4. 先恢复服务，再慢慢复盘根因

这在 agent workflow 里非常常见。

因为很多问题不是 prompt 本身坏了，而是：

- 旧 session
- 旧 fallback
- 旧 cron metadata
- 旧运行状态

把一条干净 job 拉起来，常常是最快恢复业务的方法。
