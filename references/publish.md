# AI Daily Digest 发布前 Checklist

这份文档是给准备把技能发到 GitHub / 对外分享的人看的。

目标不是“看起来完整”，而是：

- 别人看得懂
- 别人装得上
- 别人第一次试跑不容易踩坑

---

## 1. 发布前先确认这几个问题

### 1.1 这个仓库里有没有明显的本地私货

重点检查：

- 写死的本地绝对路径
- 私有 API base URL
- 私有 token / key / chat id
- 只在你自己机器上成立的文件路径
- 只为一次性排障写的临时脚本

判断标准很简单：

> 如果别人 clone 下来后，看到这段内容会问“这是什么”“我这里没有这个”，那它大概率就不该直接作为公共内容暴露。

### 1.2 文档是不是按“第一次接手的人”写的

别用“我们都知道”的口吻写。

要假设对方不知道：

- 你的目录结构
- 你之前踩过哪些坑
- 你默认用的是哪个消息渠道
- 你这套 workflow 为什么要拆成这么多步

### 1.3 先保证手动能跑通，再谈自动化

对外分享时，永远优先保证：

- 手动试跑路径清晰
- 结果质量可验证
- 出问题时可定位

不要把“自动 cron + 自动投递”写成唯一入口。

---

## 2. 建议公开的内容

建议保留：

- `SKILL.md`
- `references/setup.md`
- `references/workflow.md`
- 和主流程强相关、可以复用的脚本
- 必要的静态资源（例如示意图）

建议公开内容满足这个标准：

- 别的龙虾拿到后真能用
- 别的主人看完后能理解它值不值得装

---

## 3. 建议不要直接公开的内容

下面这些东西，如果没有进一步整理，通常不建议直接公开：

- 本地调试脚本
- 只对当前环境成立的临时脚本
- 写死本地配置路径的脚本
- 跑一次排障时留下的临时辅助文件
- 带真实投递目标、真实账号信息的配置
- 大量历史 output 运行产物

一句话：

> 技能仓库应该公开“能力”，不是公开“你的运行现场”。

---

## 4. 对 AI Daily Digest 这条技能的特别建议

### 4.1 先把“可复用主路径”讲清楚

别人最需要知道的不是所有实验历史，而是：

- 这条技能能干嘛
- 标准 workflow 是什么
- 怎么手动跑一遍
- 如何接 cron
- 出问题时先看哪层

### 4.2 不要把产物目录当成文档

`output/` 很有用，但它更像“运行现场样本”，不是技能说明本体。

建议：

- 文档里可以引用它
- 但不要让别人把理解技能的主要负担放在翻 output 上

### 4.3 公开版要强调“可替换”

对外分享时，建议一直强调：

- 模型可替换
- 投递目标可替换
- RSS 源可替换
- cron 可替换

这样别人更容易接受这不是“你私有工作流的硬编码备份”，而是一条能迁移的技能。

---

## 5. 发布前自检问题

在真正推 GitHub 前，建议自问：

- 第一次接手的人能在 10 分钟内理解这技能的价值吗？
- 第一次接手的人能在 20~30 分钟内完成一次手动试跑吗？
- 文档有没有把“主人需要准备什么”讲清楚？
- 文档有没有把“龙虾接手时该怎么问”讲清楚？
- 脚本里有没有本地路径 / 私有配置残留？
- 仓库里有没有明显不该公开的 output / 临时文件？

如果这些问题里有两个以上答不上来，最好先别急着发。

---

## 6. 推荐发布顺序

推荐按这个顺序发布：

1. 先整理文档
2. 再清理本地私货和临时脚本
3. 再决定哪些脚本值得公开
4. 最后才 push GitHub

不要反过来。

因为“先发出去再补文档”通常会让仓库越来越像你自己的工作台，而不是一个可复用技能。

---

## 7. 一句话版发布原则

如果你只能记住一句话，就记这个：

> 对外分享 skill，不是把你本地工作目录发出去，而是把“别人也能接住并跑起来的能力”发出去。
